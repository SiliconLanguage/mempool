### [04-01-2026 / Phase 0] — Banshee Emulator Unavailable: LLVM 12 Dependency Wall

*   **The Failure State:**  
    `make run-banshee` fails immediately — the `banshee` binary does not exist on the system. Banshee is an LLVM-based RISC-V emulator developed at ETH Zurich. Its build requires a specific LLVM 12 toolchain (`llvm-12-dev`, `libclang-12-dev`) which is not available in the Ubuntu 24.04 package repositories. Ubuntu 24.04 ships LLVM 18; LLVM 12 packages were dropped after Ubuntu 22.04.

*   **The Root Cause:**  
    Banshee's Rust `Cargo.toml` depends on the `llvm-sys` crate pinned to LLVM 12.x. The crate performs compile-time linking against `libLLVM-12.so` and `libclang-12.so`. These shared libraries are not binary-compatible across major LLVM versions. There is no trivial `apt install` path on Ubuntu 24.04, and building LLVM 12 from source is a multi-hour endeavor that falls outside the scope of a data plane bring-up.

    The Banshee source tree exists at `phase0_c/banshee-src/` (a git submodule), but cannot be compiled without resolving this dependency chain.

*   **The Resolution / Bypass:**  
    **Pivot to Spike (Option C).** The Spike RISC-V ISA Simulator (`riscv-isa-sim`) is already installed at `install/riscv-isa-sim/bin/spike` as part of the standard MemPool toolchain. Spike provides functional ISA-level simulation (no cycle accuracy) with multi-hart support, which is sufficient for validating RVWMO atomic correctness.

    A full `run-spike` target was added to the Makefile:
    ```makefile
    SPIKE ?= /home/dragonix/mempool/install/riscv-isa-sim/bin/spike
    SPIKE_NUM_CORES ?= 2

    run-spike: $(APP)_spike_stripped.elf
        @echo "Spike will loop indefinitely (no HTIF _eoc). Use Ctrl-C or wait for timeout."
        -timeout 10 $(SPIKE) --isa=rv32ima -p$(SPIKE_NUM_CORES) \
          -m0x10000:0x40000,0x40000000:0x100000,0x80000000:0x400000 \
          $<
    ```

*   **Architectural Takeaway:**  
    The data plane validation strategy must not be gated on a single emulator. Spike validates ISA-level correctness (atomic ordering, memory faults, trap behavior). Banshee would add cycle-level timing. RTL simulation (Verilator/QuestaSim) is the gold standard. By decoupling functional correctness from performance modeling, we can iterate on the lock-free queue design without waiting for Banshee's LLVM dependency to be resolved.

### [04-01-2026 / Phase 0] — GNU ld 2.42 RWX Segment Warning Treated as Error

*   **The Failure State:**  
    ```
    riscv64-unknown-elf-ld: warning: mempool_ipc_phase0.elf has a LOAD segment with RWX permissions
    ```
    While technically a warning (not an error), this pollutes build output and in some CI configurations with `-Werror` linker wrappers, it causes a hard build failure.

*   **The Root Cause:**  
    GNU ld 2.42 introduced stricter diagnostics for ELF segments that request Read+Write+Execute permissions simultaneously. MemPool's `link.ld` places `.l1_seq` and `.l1` as `(NOLOAD)` sections into the `l1` memory region, which is declared as `l1 (R)` in `arch.ld.c`. The linker collapses all L1 sections into a single LOAD segment. Because `.l1` contains `.bss` (writable) and the region is marked readable, the result is an RWX LOAD program header.

    This is a false positive for bare-metal: there is no MMU or page table — the L1 TCDM is physically mapped scratchpad memory with no permission bits. The warning is purely an artifact of the ELF container format.

*   **The Resolution / Bypass:**  
    Added `-Wl,--no-warn-rwx-segments` to LDFLAGS:
    ```makefile
    LDFLAGS = -L. -T $(RUNTIME_DIR)/link.ld -nostdlib -nostartfiles -lgcc \
              -Wl,--no-warn-rwx-segments
    ```
    This suppresses the warning at the linker level. The same flag is also applied to the Spike link step.

*   **Architectural Takeaway:**  
    Bare-metal RISC-V targets will increasingly collide with security hardening in newer GNU toolchains. The RWX warning is designed for Linux userspace ELFs where W+X is a code injection vector. For MemPool's physically-addressed TCDM, this check is meaningless. Future phases should expect similar friction from toolchain updates and document the suppression explicitly rather than silently ignoring warnings.

### [04-01-2026 / Phase 0] — Unexpanded CPP Macros in Linker Script (`BOOT_ADDR`, `L1_BANK_SIZE`)

*   **The Failure State:**  
    ```
    riscv64-unknown-elf-ld: arch.ld: unexpected token `BOOT_ADDR' in MEMORY region
    riscv64-unknown-elf-ld: arch.ld: unexpected token `L1_BANK_SIZE' in MEMORY region
    ```
    The linker rejects the generated `arch.ld` because it contains literal C macro names instead of numeric values.

*   **The Root Cause:**  
    `runtime/arch.ld.c` is a linker script template processed by the C preprocessor (`gcc -E -P -x c`). It relies on macros like `BOOT_ADDR`, `L1_BANK_SIZE`, `L2_BASE`, `L2_SIZE`, `NUM_CORES`, `BANKING_FACTOR`, and `SEQ_MEM_SIZE` being defined via `-D` flags in CFLAGS.

    The original Phase 0 Makefile's CFLAGS included `NUM_CORES`, `BANKING_FACTOR`, and `SEQ_MEM_SIZE` (copied from MemPool's top-level build) but was **missing** `-DBOOT_ADDR=...` and `-DL1_BANK_SIZE=...`. These macros are normally injected by MemPool's top-level `Makefile` → `config/mempool.mk` chain, which Phase 0's standalone Makefile does not source.

    The CPP preprocessor silently passes through undefined tokens in non-`#if` contexts — the undefined `BOOT_ADDR` macro appears inside a linker script `ORIGIN = BOOT_ADDR` expression, not a `#define` guard. The preprocessor outputs it as literal text, and the linker chokes.

*   **The Resolution / Bypass:**  
    Added the missing macros to CFLAGS, with values matching `config/mempool.mk`:
    ```makefile
    CFLAGS = ... \
             -DBOOT_ADDR=0x00001000 \
             -DL1_BANK_SIZE=1024
    ```

    The values were determined by inspecting `config/mempool.mk` and `runtime/arch.ld.c`:
    - `BOOT_ADDR = 0x00001000` → MemPool's hardware boot ROM address
    - `L1_BANK_SIZE = 1024` → 1 KB per L1 TCDM bank (256 cores × 4 banks × 1024 = 1 MB total)

*   **Architectural Takeaway:**  
    MemPool's build system uses CPP-as-linker-template-engine. Any standalone Makefile that bypasses the top-level build must explicitly define **every** macro consumed by `arch.ld.c`. The CPP preprocessor gives no warning for undefined macros used outside `#if`/`#ifdef` — they silently become literal text. This is a category of silent failure unique to the "linker script as C source" pattern. A future improvement would be to add `#ifndef BOOT_ADDR / #error` guards to `arch.ld.c` itself.

### [04-01-2026 / Phase 0] — `.l1_seq` Section Overflow: SEQ_MEM_SIZE × NUM_CORES > L1 Capacity

*   **The Failure State:**  
    ```
    riscv64-unknown-elf-ld: mempool_ipc_phase0.elf section `.l1_seq' will not fit in region `l1'
    riscv64-unknown-elf-ld: region `l1' overflowed by 1048576 bytes
    ```

*   **The Root Cause:**  
    The linker script allocates sequential memory as `NUM_CORES × SEQ_MEM_SIZE` bytes at the start of L1. With the original values:
    - `SEQ_MEM_SIZE = 8192` (8 KB per core)
    - `NUM_CORES = 256`
    - Sequential region = 256 × 8192 = **2 MB**
    - L1 total = `NUM_CORES × BANKING_FACTOR × L1_BANK_SIZE` = 256 × 4 × 1024 = **1 MB**

    2 MB > 1 MB → overflow by exactly 1 MB (1,048,576 bytes), matching the error.

    The sequential region (`__seq_start` to `__seq_end`) reserves per-core private memory within the interleaved L1 TCDM. It is used for stacks and per-tile hardware queue buffers. The `.l1_seq` section in `link.ld` advances the location counter to `__seq_end`, reserving this space before the interleaved `.l1` section begins.

*   **The Resolution / Bypass:**  
    Reduced `SEQ_MEM_SIZE` from 8192 to 2048 (matching `STACK_SIZE`) and updated `LOG2_SEQ_MEM_SIZE` accordingly:
    ```makefile
    -DSEQ_MEM_SIZE=2048 -DLOG2_SEQ_MEM_SIZE=11
    ```
    New sequential region: 256 × 2048 = 512 KB < 1 MB L1. ✓

    This is safe because our Phase 0 SPSC test does not use per-core sequential allocation beyond the stack. The `ipc_queue` and `boot_barrier` are placed in the interleaved `.l1` section, not the sequential region.

*   **Architectural Takeaway:**  
    The constraint `NUM_CORES × SEQ_MEM_SIZE ≤ NUM_CORES × BANKING_FACTOR × L1_BANK_SIZE` simplifies to `SEQ_MEM_SIZE ≤ BANKING_FACTOR × L1_BANK_SIZE` = 4 × 1024 = 4096 bytes per core. Any `SEQ_MEM_SIZE > 4096` will overflow L1 on the 256-core config. Future phases that need larger per-core buffers must either increase `L1_BANK_SIZE` or move data to L2.

### [04-01-2026 / Phase 0] — Spike ELF Loader Faults on NOLOAD Segments at Address 0x0

*   **The Failure State:**  
    ```
    trap_store_access_fault
    ```
    Spike crashes immediately during ELF loading (before any instruction executes). The fault occurs inside Spike's C++ `sim_t::write_chunk()` method, not from a guest instruction.

*   **The Root Cause:**  
    MemPool's `link.ld` defines `.l1_seq` and `.l1` as `(NOLOAD)` sections mapped to L1 at address `0x00000000`. Despite the NOLOAD attribute, the GNU linker emits a LOAD program header for the L1 region (because the memory region is declared `l1 (R)` in `arch.ld.c`). This LOAD segment has:
    - `VirtAddr = 0x00000000`
    - `FileSiz = 0` (no file content — correct for NOLOAD)
    - `MemSiz = 0x310C0` (the queue + barrier allocation)

    Spike's ELF loader (in `toolchain/riscv-isa-sim/riscv/sim.cc`, line ~322) processes all LOAD segments and zero-fills the region from `FileSiz` to `MemSiz` using `write_chunk()`, which calls `debug_mmu->store_uint64()`. This MMU store resolves virtual address `0x0` against Spike's configured physical memory regions. Address `0x0` is not in any `-m` region, causing a `trap_store_access_fault` inside the host-side ELF loader.

    This is a Spike-specific issue. Real hardware and RTL simulation handle NOLOAD correctly because the L1 TCDM is always physically present at address 0.

*   **The Resolution / Bypass:**  
    Strip the NOLOAD sections from the ELF before passing it to Spike:
    ```makefile
    $(APP)_spike_stripped.elf: $(APP)_spike.elf
        $(OBJCOPY) --remove-section=.l1_seq --remove-section=.l1 $< $@
    ```
    The stripped ELF retains `.text`, `.rodata`, `.data`, and `.htif` in L2 — everything Spike needs to execute. The L1 regions are uninitialized scratchpad; their contents are written at runtime by `main()`.

*   **Architectural Takeaway:**  
    Spike's ELF loader walks the MMU for every byte of every LOAD segment, including zero-fill regions. This means NOLOAD sections that happen to generate a LOAD program header will fault if their target address is outside Spike's `-m` regions. The `objcopy --remove-section` workaround is robust: it removes the sections entirely from the ELF container, eliminating the LOAD segment. This is a permanent pattern for any Spike target that uses L1 NOLOAD sections.

### [04-01-2026 / Phase 0] — 256-Hart Barrier Deadlock on 2-Hart Spike

*   **The Failure State:**  
    Spike hangs indefinitely. Trace shows both harts stuck in a tight loop:
    ```
    amoadd.w.aq  zero, a4, (a5)   # increment boot_barrier
    lw           a3, 0(a5)         # load boot_barrier
    bltu         a4, a3, ...       # branch if barrier_count < NUM_CORES
    ```
    Hart 0 and Hart 1 each increment `boot_barrier` to 2, but `hw_barrier()` spins until `boot_barrier >= NUM_CORES` which equals 256 (compiled into the binary as an immediate).

*   **The Root Cause:**  
    The main build compiles with `-DNUM_CORES=256`. Using this same binary on Spike with `-p2` means only 2 harts execute, but the barrier expects 256 participants. The barrier will never complete.

    This is not just a barrier issue — `mempool_get_core_count()` returns the compile-time constant `NUM_CORES`, so any code that iterates over cores or distributes work will also be wrong.

*   **The Resolution / Bypass:**  
    Created a separate Spike compilation pipeline with `SPIKE_CFLAGS` that overrides core-count macros:
    ```makefile
    SPIKE_NUM_CORES ?= 2
    SPIKE_CFLAGS = $(subst -DNUM_CORES=256,-DNUM_CORES=$(SPIKE_NUM_CORES), \
                   $(subst -DLOG2_NUM_CORES=8,-DLOG2_NUM_CORES=1, \
                   $(subst -DLOG2_NUM_CORES_PER_GROUP=6,-DLOG2_NUM_CORES_PER_GROUP=0, \
                   $(subst -DL1_BANK_SIZE=1024,-DL1_BANK_SIZE=32768, \
                   $(CFLAGS)))))
    ```
    All Spike-specific objects (`spike_src/main.o`, `spike_crt0.o`, `spike_arch.ld`) are compiled with `SPIKE_CFLAGS`. The `$(subst)` chain transforms CFLAGS in-place without duplicating the entire flag string.

*   **Architectural Takeaway:**  
    MemPool's runtime bakes hardware topology into compile-time constants (`NUM_CORES`, `LOG2_*`, `L1_BANK_SIZE`). Any emulator that runs fewer harts than the compiled core count requires a separate compilation. This is not a bug — it is a deliberate bare-metal design: no dynamic discovery, no device tree, no runtime enumeration. The cost is that every target configuration needs its own object files.

### [04-01-2026 / Phase 0] — `INCLUDE arch.ld` Resolves from CWD, Not `-L` Path

*   **The Failure State:**  
    After creating `spike_arch.ld` (2-core memory map), the Spike link step still produced an ELF with `__l1_end = 0x00100000` (1 MB = 256-core layout) instead of `__l1_end = 0x00050000` (256 KB = 2-core layout). The linker consumed the wrong `arch.ld`.

*   **The Root Cause:**  
    The runtime `link.ld` contains `INCLUDE arch.ld`. GNU ld resolves `INCLUDE` directives by searching:
    1. The current working directory (CWD)
    2. The `-L` library search paths

    The main build generates `./arch.ld` (256-core) in the CWD. When the Spike link step uses `-T ../../runtime/link.ld`, the `INCLUDE arch.ld` in that file finds `./arch.ld` in CWD **before** any `-L` search path. There is no way to make `-L` take priority over CWD for `INCLUDE` resolution.

    Even if `spike_arch.ld` were named `arch.ld` in a subdirectory with `-Lspike_build`, the CWD copy would still win.

*   **The Resolution / Bypass:**  
    Created `spike_link.ld` — a copy of `runtime/link.ld` with one change:
    ```diff
    - INCLUDE arch.ld
    + INCLUDE spike_arch.ld
    ```
    The Spike link step uses `-T spike_link.ld` which includes `spike_arch.ld` by exact name, avoiding any ambiguity. The file `spike_arch.ld` is generated alongside `spike_link.ld` in the CWD.

*   **Architectural Takeaway:**  
    GNU ld's `INCLUDE` directive CWD-first resolution is a trap when multiple configurations share the same linker script template. The only reliable fix is to give each generated linker fragment a unique filename. Do not rely on `-L` search order for `INCLUDE` disambiguation.

### [04-01-2026 / Phase 0] — L1 TCDM Too Small for HardwareQueue at 2-Core Config

*   **The Failure State:**  
    ```
    riscv64-unknown-elf-ld: mempool_ipc_phase0_spike.elf section `.l1' will not fit in region `l1'
    ```
    The Spike build with `NUM_CORES=2` and `L1_BANK_SIZE=1024` yields:
    - L1 = 2 × 4 × 1024 = **8 KB**
    - `HardwareQueue` = 1024 slots × 64 bytes + 2 × 64 bytes = **65,664 bytes (~65 KB)**

    65 KB >> 8 KB → overflow.

*   **The Root Cause:**  
    The L1 size formula `NUM_CORES × BANKING_FACTOR × L1_BANK_SIZE` scales linearly with core count. At 256 cores, L1 = 1 MB (comfortable). At 2 cores, L1 = 8 KB (cannot hold even a single `HardwareQueue`).

    The queue's `RING_SIZE=1024` with 64-byte cacheline-aligned slots is sized for the full MemPool interconnect bandwidth. It does not scale down with core count.

*   **The Resolution / Bypass:**  
    Override `L1_BANK_SIZE` in `SPIKE_CFLAGS` to compensate:
    ```makefile
    $(subst -DL1_BANK_SIZE=1024,-DL1_BANK_SIZE=32768, ...)
    ```
    New L1 = 2 × 4 × 32768 = **256 KB** — comfortably fits the 65 KB queue, the sequential region (2 × 2048 = 4 KB), and stacks.

    This does not affect the main 256-core build, which retains `L1_BANK_SIZE=1024`.

*   **Architectural Takeaway:**  
    When reducing core count for emulation, L1 capacity drops proportionally. Fixed-size data structures (queues, DMA descriptors, weight buffers) that were sized for the full system may not fit. The simplest fix is to inflate `L1_BANK_SIZE` for the reduced-core config. An alternative would be to scale `RING_SIZE` with `NUM_CORES`, but that changes the data plane behavior under test.

### [04-01-2026 / Phase 0] — Spike Internal Boot ROM Collision at 0x1000

*   **The Failure State:**  
    ```
    trap_store_access_fault at mepc=0x80000xxx
    ```
    After fixing all previous issues, Spike faults when Hart 0 executes the first `amoadd.w` to `boot_barrier`. Trace shows the target address is `0x00001000`.

*   **The Root Cause:**  
    Spike places its internal boot ROM at `DEFAULT_RSTVEC = 0x1000`. This ROM contains the reset vector that jumps to the ELF entry point. It is **read-only** and **not backed by DRAM** — it is a hardcoded ROM inside Spike's C++ `sim_t` object.

    With L1 ORIGIN at `0x00000000` (from `arch.ld.c`), the generated symbol layout is:
    - `__seq_end = 0x1000` (2 cores × 2048 bytes)
    - `boot_barrier` lands at `0x1000` (first symbol in `.l1` after `.l1_seq`)

    Address `0x1000` is Spike's ROM. A store to ROM → `trap_store_access_fault`.

    This collision does not occur on real MemPool hardware because there is no ROM at `0x1000` — the boot address is a hardware reset vector, not a memory-mapped ROM.

*   **The Resolution / Bypass:**  
    Post-process `spike_arch.ld` to relocate L1 ORIGIN above the ROM:
    ```makefile
    spike_arch.ld: $(RUNTIME_DIR)/arch.ld.c
        $(CC) $(SPIKE_CFLAGS) -E -P -x c $< -o $@
        sed -i 's/ORIGIN = 0x00000000/ORIGIN = 0x00010000/' $@
    ```
    New layout:
    - L1 ORIGIN = `0x00010000` (64 KB, well above Spike's 4 KB ROM)
    - `boot_barrier` = `0x00021000`
    - `ipc_queue` = `0x00021040`
    - `__l1_end` = `0x00050000`

    Spike's `-m` flag updated to match: `-m0x10000:0x40000,...`

*   **Architectural Takeaway:**  
    Spike's `DEFAULT_RSTVEC` at `0x1000` is an immutable architectural constant. Any bare-metal binary that places writable data below `0x10000` risks colliding with this ROM. The 64 KB offset (`0x10000`) provides a safe margin. This constraint is Spike-specific and does not apply to RTL simulation or real hardware where L1 TCDM at address 0 is valid.

### [04-01-2026 / Phase 0] — Spike Cannot Terminate: `_eoc` Writes to Control Register, Not HTIF

*   **The Failure State:**  
    Spike runs both harts to completion (zero exceptions), but loops forever. The trace shows `_eoc` executing:
    ```asm
    li    t0, 0x40000000     # CONTROL_REGISTER_OFFSET + EOC_REG
    slli  t1, a0, 1
    addi  t1, t1, 1
    sw    t1, 0(t0)          # Write EOC status to control reg
    la    ra, __rom_start    # Load boot ROM address
    ret                      # Jump to boot ROM → restart
    ```
    The write to `0x40000000` is absorbed by Spike's DRAM backing (control register region is configured as a memory region via `-m`). The `ret` jumps to `__rom_start = 0x1000` which is Spike's reset vector, restarting execution from `_start`.

*   **The Root Cause:**  
    MemPool's `_eoc` handler in `crt0.S` signals completion by writing to a hardware control register. In the real MemPool system, this triggers an end-of-computation interrupt to the host controller. Spike has no such hardware — its termination mechanism is HTIF (Host-Target Interface), which requires a write to the `tohost` symbol with a specific payload.

    Although `spike_htif.S` provides `tohost` and `fromhost` symbols (required for Spike to even start), `_eoc` does not use them. Modifying `crt0.S` to add HTIF termination would change the shared runtime code.

*   **The Resolution / Bypass:**  
    Accepted as a known limitation. The `run-spike` target wraps execution with `timeout 10`:
    ```makefile
    run-spike: $(APP)_spike_stripped.elf
        @echo "Spike will loop indefinitely (no HTIF _eoc). Use Ctrl-C or wait for timeout."
        -timeout 10 $(SPIKE) --isa=rv32ima -p$(SPIKE_NUM_CORES) ...
    ```
    The `-` prefix tells `make` to ignore the exit code 124 from `timeout`. Functional correctness is verified via Spike's instruction trace (`-l` flag) rather than program exit status.

*   **Architectural Takeaway:**  
    MemPool's boot code is written for real hardware, not emulators. `_eoc` → control register → host interrupt is the correct hardware path. Spike's HTIF is a fundamentally different termination mechanism. Bridging the two would require either: (a) patching `_eoc` to also write HTIF (invasive, changes shared code), or (b) a Spike plugin that traps writes to `0x40000000` and converts them to HTIF exits. For Phase 0, timeout-based execution with trace verification is sufficient.
